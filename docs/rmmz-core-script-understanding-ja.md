# RPGツクールMZ コアスクリプト理解メモ（プラグイン開発向け）

この文書は、RPGツクールMZプラグイン開発時に「どこを拡張すべきか」を判断しやすくするための実践メモです。  
厳密な内部実装はバージョン差分で変わるため、ここでは**設計の骨格**と**安全な介入パターン**を重視しています。

---

## 1. コアスクリプト全体像

MZの標準スクリプトは概ね次の責務分割になっています（ファイル名は代表例）。

- `rmmz_core.js`
  - 低レベル基盤（ユーティリティ、描画基盤、入力、オーディオ、Bitmap/Sprite/TilingSprite/Window など）
- `rmmz_managers.js`
  - 各種マネージャ（`DataManager`, `SceneManager`, `ImageManager`, `AudioManager`, `ConfigManager`, `PluginManager` など）
- `rmmz_objects.js`
  - ゲームロジックの実体（`Game_*` クラス群）
- `rmmz_scenes.js`
  - シーン遷移と画面単位の制御（`Scene_Boot`, `Scene_Map`, `Scene_Battle`, `Scene_Menu` など）
- `rmmz_sprites.js`
  - 戦闘やマップの描画スプライト系（`Spriteset_*`, `Sprite_*`）
- `rmmz_windows.js`
  - UIウィンドウ（`Window_*`）

プラグイン開発では、次の対応関係を意識すると実装ミスが減ります。

- 表示を変える: `Window_*` / `Sprite_*` / `Scene_*`
- 入力や操作導線を変える: `Scene_*` / `Window_*` / `Input` / `TouchInput`
- ゲームルールを変える: `Game_*` / `BattleManager` / `DataManager`
- 設定や永続化を変える: `ConfigManager` / `DataManager`

---

## 2. 更新ループ（毎フレーム何が起きるか）

MZは `SceneManager` 主導で更新ループが進みます。ざっくり次の流れです。

1. 現在シーンの `update`
2. 子要素（ウィンドウ・スプライト）の `update`
3. 描画更新（PIXI側）
4. 入力状態の更新（`Input`, `TouchInput`）

**実務上のポイント**

- 毎フレーム処理は軽量化する（重い走査はキャッシュ）
- シーン単位の処理は `Scene_*` に寄せる
- UI更新は `Window_*` 側に寄せる（責務分離）

---

## 3. データ読み込みとゲーム開始フロー

代表的には次の順で進みます。

1. `Scene_Boot` が開始
2. `DataManager` がDBデータ（`$dataActors` など）をロード
3. タイトル画面 or ニューゲーム開始
4. ニューゲーム時に `Game_*` のランタイム実体（`$gameParty`, `$gameMap` など）を初期化

### 重要なグローバル実体

- DB由来（静的寄り）: `$data*`
- セーブ対象ランタイム: `$game*`
- システム単位: `$gameSystem`, `$gameSwitches`, `$gameVariables`

**設計指針**

- プラグイン設定の「定義値」はパラメータへ
- プレイ中に変化する値は `$gameSystem` などのセーブ対象へ
- DBメタ（ノートタグ）を使う場合はロード後に整形してキャッシュ

---

## 4. クラス層の押さえどころ

### 4.1 Scene

- 画面単位の制御ハブ
- `create`, `start`, `update`, `terminate` が主要ライフサイクル
- 画面遷移やウィンドウ配置の起点

### 4.2 Window

- UI表示とUI入力
- `Window_Base` / `Window_Selectable` / `Window_Command` 系を基底に拡張
- `refresh` の呼び出しタイミングを管理し、無駄再描画を避ける

### 4.3 Sprite / Spriteset

- 視覚表現レイヤ
- マップや戦闘のエフェクト拡張は `Sprite_*` と `Spriteset_*` が中心

### 4.4 Game_*

- ゲーム状態・ルール本体
- 永続化対象になりやすく、仕様変更の影響範囲が大きい

---

## 5. プラグイン介入ポイント（優先順）

### A. Plugin Command（MZ標準）

- イベント経由で実行しやすい
- 引数が明確で、制作者向け導線を作りやすい

### B. 既存メソッドのエイリアス拡張

最も一般的な拡張手法です。オリジナルを保存してから拡張します。

```js
const _Game_Party_gainGold = Game_Party.prototype.gainGold;
Game_Party.prototype.gainGold = function(amount) {
  _Game_Party_gainGold.call(this, amount);
  // 追加処理
};
```

### C. 新規クラス/ユーティリティ導入

- 影響範囲を閉じたい場合に有効
- 既存プロトタイプ汚染を最小化できる

---

## 6. 安全なオーバーライド規約（実践）

1. 元メソッド参照を保持する（エイリアス）
2. `this` を崩さない（`call/apply`）
3. 戻り値契約を維持する
4. 条件付きで早期returnするときは副作用漏れを確認する
5. 競合しやすいメソッドにはガードを入れる

競合回避例:

- 名前空間を1つに集約（例: `MyPlugin` オブジェクト）
- 二重パッチ防止フラグを持つ
- 他プラグイン順序に依存する場合はREADMEに明記

---

## 7. セーブ/ロードと永続化の基本

永続化設計の原則:

- 設定値（ユーザー設定）: `ConfigManager`
- プレイデータ: `$gameSystem` など `Game_*` 系
- 一時キャッシュ: セーブ対象外（再計算可能な値）

注意点:

- セーブデータ互換を壊す変更は移行処理を検討
- 既存セーブに新規プロパティ追加時はデフォルト補完を入れる

---

## 8. パフォーマンスと安定性チェック

- 毎フレームで `filter/map/reduce` を多用しない
- 画像ロードは `ImageManager` を使って管理する
- `refresh` 連打を避け、dirtyフラグで制御する
- null/undefinedガードを徹底する（戦闘・遷移時の未初期化を想定）

---

## 9. デバッグ観点（再現性重視）

不具合調査時に最低限残す情報:

- 発生シーン（Map/Battle/Menu）
- 直前操作
- プラグイン順序
- セーブデータ再現可否（NewGame限定か）
- コンソールエラー全文（スタック）

切り分けの順序:

1. 当該プラグイン単体で再現確認
2. 競合候補を順次有効化
3. エイリアス先/後でログ採取
4. セーブ有無で差分確認

---

## 11. 今後この記録を育てる方針

この文書は「理論メモ」で終わらせず、次のログを追記して精度を上げるのが有効です。

- 実際に起きた不具合と原因
- 競合したプラグイン組み合わせ
- 採用した回避策（コード断片）
- パフォーマンス改善前後の計測値

運用としては、機能追加ごとに「どの章に関係したか」を1行追記すると、開発資産として価値が上がります。
